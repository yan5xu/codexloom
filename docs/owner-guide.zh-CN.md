# CodexLoom Owner Guide（简体中文）

> 面向长期使用 Codex Agent Team 的高级个人 Owner 的评审草稿。

**简体中文** · [English translation](owner-guide.md)

> **本文是 Owner Guide 的权威版本与主要评审文本。**
> [owner-guide.md](owner-guide.md) 是它的英文译本，用于对外阅读。
> 两者含义出现分歧时，以本文为准，并应回头修订英文译本。

本草稿中的“当前行为”陈述已对照 `origin/main` 的 `81696e1` 提交核验。
Topic 与 Trigger 被单独标记为开发构建行为，因为它们不属于该基线。

CodexLoom 帮助一个人建立、使用、协调和调整一支长期在岗的 Codex Agent 团队。
Codex 仍然是 Agent Runtime；Loom 在其之上增加稳定身份、持续责任、通信、
受治理的外部角色，以及关于团队如何运转的证据。

本指南围绕 Owner 的工作组织，而不是围绕 Loom 的数据对象或基础设施。

## 如何阅读本指南

本指南区分五类陈述：

- **产品原则**（Product principle）：CodexLoom 意图守住的产品边界。
- **当前行为**（Current behavior）：当前 `main` 分支已经提供的行为。
- **已验证实践**（Validated practice）：被反复真实使用支持的工作方式。
- **当前建议**（Current recommendation）：当前有用、但可能随更多实践改变的指引。
- **假设**（Hypothesis）：仍需证据才能成为指引的想法。

仅存在于开发构建中的能力会被显式标注。它们不应被当作当前 `main` 分支的承诺。

## 核心思路

### 从持续发生的工作出发，而不是从组织架构图出发

第一个问题不是“我应该创建多少个 Agent？”，而是：

> 哪些工作需要一个持续的负责人、持续的上下文和专业判断？

一次性任务可以保持为任务；可复用的固定流程可能属于 Skill。当相关工作反复
受益于同一个身份、工作历史、责任与判断时，长期 Agent 才开始有价值。

**产品原则：** Agent 是一种持久的工作关系，而不是加进架构图里的一个职位名称。

### 先让一个 Agent 承担工作，直到稳定边界自己出现

**当前建议：** 从最小可用配置开始——一个 Agent，一份收窄的 Profile，一项真实工作。

它的主要 Thread 保留工作轨迹，Profile 则声明那些应当保持稳定的责任。

不要仅仅因为某项任务很大或缺少某个工具就拆出新的 Agent。继续观察真实工作。
当证据显示单个 Agent 正在承载互不相容的上下文、彼此不同的专业判断、持续的
负载，或一项值得独立负责人的责任时，分化才变得有意义。

### 组织跟随分化而来

当责任被分给多个长期 Agent 之后，通信与组织才成为必要。这时 Loom 帮助回答：

- 谁仍然对整体负责？
- 每个稳定的 Domain 由哪个 Agent 拥有？
- 哪些关系是层级责任，哪些是稳定的跨领域接口？
- 一个问题、结果或升级应该回到哪里？

**当前建议：** 当组织记录的是真实发生的责任分工，而不是预测一个理想的未来结构时，
它更可靠。

## Owner 的工作节奏

CodexLoom 有多个工作界面，因为 Owner 使用它们的频率不同。

| 频率 | 界面 | 主要问题 |
|---|---|---|
| 一天中的大部分时间 | Agent workspace | 我正在和哪个长期 Agent 一起工作？ |
| 被请求时 | Needs You | 哪个事实、选择或授权需要我？ |
| 周期性 | Overview | 团队运转是否正常，什么值得进一步调查？ |
| 责任发生变化时 | Team | 谁负责什么，已声明的关系是否仍然符合真实工作？ |
| 外部角色发生变化时 | External | 哪个 Agent 可以在哪个 Conversation 中以哪个身份行动？ |
| 维护时 | Settings | 本地 Runtime、Connector、备份和远程访问是否健康？ |

Agent workspace 是默认界面。Overview 是观察界面，不是公司仪表盘；Settings 是
维护界面，不是日常工作的中心。

## 建立你的第一个长期 Agent

### 1. 选择一项持续发生的责任

使用已经在反复发生的工作。例如维护一个产品领域、进行持续研究、运营一项内容
实践，或支持一段长期客户关系。

请自问：

- 这个 Agent 应该跨任务记住什么？
- 哪种判断应该随着反复工作而变好？
- 什么应该自然地进入或离开它的责任范围？
- 谁来判断它的工作是否正确？

如果答案只描述了今天的交付物，那么应该创建一个任务，而不是一个新的长期 Agent。

### 2. 写出最小可用的 Profile

一份起步 Profile 需要三个部分：

- **Identity：** 在这段工作关系中，这个 Agent 是谁。
- **Domain：** 它长期拥有的对象与专业判断。
- **Scope：** 它可以做什么、应该拒绝什么，以及何时应该回到 Owner 或另一个
  负责的 Agent。

把 Profile 当作一个可检验的假设。它应该具体到足以安全开始，同时在真实工作
之后仍然容易修订。

### 3. 交给它一项真实的首次工作

创建了 Agent 记录，并不证明这个角色成立。用一项真实工作来验证：这个 Agent
能否访问必要的工具与 Skill、产出有用的结果、保持连续性，并在预期边界处停下。

### 4. 把相关工作持续交给同一个 Agent

把相关工作交回同一个 Agent 和同一条 Thread。纠正、术语、约束、产物和决策由此
成为它持续工作轨迹的一部分。只有当某项变化应当成为长期责任时才更新 Profile，
不要为每一条临时指令修改它。

## 日常与 Agent 一起工作

正常的循环是：

1. 在 Agent workspace 中给出目标、问题或具体工作。
2. Agent 在自己的 Thread 中工作、使用工具，并在需要其他 Agent 的判断时与其联系。
3. 普通的内部回复和外部交互回到直接负责的 Agent，而不是回到 Owner。
4. 真正需要人来决定的事项出现在 Needs You。
5. Agent 把可读的结果返回给发起这项工作的那段关系。

关键区分是因果关系：

> 不要问每一条输出看起来是否重要，而要问它完成了谁的工作、这个结果应该回到哪里。

Agent Inbox 属于 Agent 自己。它可以作为负载、路由或工具问题的有用证据，但不是
Owner 的第二份任务清单。Needs You 才包含需要 Owner 采取行动的事项。

**已验证实践：** 普通的 Agent 间回复应当回到发起请求的 Agent，由它完成整合。
把每一条中间回复都发给 Owner，会重新制造出这支团队本应减少的转发负担。

## 选择正确的协调机制

不同的 Loom 机制解决不同的连续性问题。

| 需求 | 使用 | 它保留了什么 |
|---|---|---|
| 与一个 Agent 继续正常工作 | Agent Thread | 该 Agent 的工作轨迹 |
| 让一个 Agent 继续长时间运行的工作 | Goal | Runtime 的续跑与完成状态 |
| 请求另一个 Agent 的领域判断 | Message | 请求、回复、投递与因果归属 |
| 获取人的事实、选择或授权 | Needs You | 一次持久的人类决定与恢复路径 |
| 在已知时间唤醒工作 | Schedule | 基于时间的重复 |
| 在跨天、跨 Agent 之间共享一件有边界的事项 | Topic（开发构建） | 一个 Responsible Agent、限定范围的 Participants、brief、等待状态和证据链接 |
| 在外部事实变化时恢复工作 | Trigger（开发构建） | 一个受治理的、去重新核验 Provider 状态的理由 |

**当前行为：** 在本草稿所依据的 `main` 基线上，Agent Thread、Goal、Message、
Needs You 和 Schedule 均已存在。

**开发构建：** Topic 与 Trigger 正在当前本地开发构建中运行，但不在该 `main`
基线中。它们最终面向用户的说明必须在实现被集成之后重新核验。

### Message 用于直接的责任边界

当一个 Agent 需要另一个长期 Agent 的判断、工作或反馈时，它发送 Message。回复
返回给发起请求的 Agent。Owner 不应该需要转发普通的 Agent 间通信。

### Needs You 用于人的决定

只有当工作在缺少 Owner 的某个事实、选择或授权时无法负责任地继续，才使用
Needs You。说明什么被阻塞、确切的问题是什么，以及每个可选项的后果。Owner 回应
之后，原来的 Agent 应当继续同一项工作，而不是要求 Owner 重新陈述上下文。

### Topic 用于有边界的共享连续性

**开发构建：** 该机制不属于本草稿所依据的 `main` 基线。

Topic 不是共享的模型 Session，也不是项目管理看板。它是一条很薄的协调记录，用于
一件跨越多个 Turn、多天或多个 Agent 的有边界事项。它有一个 Responsible Agent、
限定范围的 Participants、一份带版本的 brief、等待条件，以及指向证据的链接。
每个 Participant 在自己的 Agent Thread 中保留详细的专业工作，并把范围内的结果
返回给 Responsible Agent。

**当前建议：** 只有当这种共享连续性在其他方式下重建成本很高时，才创建 Topic。
一次小的交接或一场会议并不自动需要一个 Topic。

### Trigger 是重新核验的理由，而不是结论

**开发构建：** 该机制不属于本草稿所依据的 `main` 基线。当前适配器只观察受支持的
GitHub pull request 与 workflow run 变化。部署、审批、Webhook 及其他 Provider
仍属于未来范围。

一次受支持的 pull request 或 workflow run 变化可以唤醒负责的 Agent。该事件是一个
去读取当前权威 Provider 状态的理由，不应被当作整体工作已经完成的证明。

## 从一个 Agent 成长为一支团队

### 一项责任可能需要拆分的证据

**假设：** 下列反复出现的信号可以支持发起一次组织调查，但没有任何单一信号能够
证明确实需要一个新的 Agent。

当以下若干信号反复出现时，可以考虑新增 Agent：

- 一个 Agent 必须同时保留互不相关的高分辨率上下文；
- 彼此不同的专业判断需要不同的负责人；
- 持续的执行时间与排队等待显示出稳定的产能边界；
- 一个 Lead 反复下钻到同一类专业实现细节中；
- 同一项责任被反复交接并在交接中丢失上下文；
- 一条独立的权限或外部身份边界需要长期保持。

这些信号启动一次调查，并不自动开出拆分的处方。

### 反对创建新 Agent 的证据

不要仅仅因为下列理由创建 Agent：

- 某一项任务格外大；
- 某个可复用流程其实可以是一个 Skill；
- 现有 Agent 缺少某个工具；
- 一个临时项目需要一名参与者；
- 加一个名字会让组织架构图看起来更完整；
- 拟设的角色只负责转发消息或状态。

### 让四类团队证据保持分离

- **Profile：** 一个 Agent 当前声明自己拥有什么。
- **Organization：** 长期的上下级责任边界。
- **Collaboration：** 独立领域之间稳定的横向接口。
- **Activity：** 限定时间范围内、真实发生过的工作证据。

Topic 可以临时集合一组 Participants，而不声明一段长期关系。Message 可以记录一次
协作，而不把它变成一条 Organization 或 Collaboration 关系。

## 给 Agent 受治理的外部角色

从 Agent 向外思考：

1. 哪个长期 Agent 负责？
2. 哪个外部身份在某个 Provider 上代表它？
3. 它在这个具体 Conversation 中拥有什么角色和权限？

一条 Conversation Membership 应当让 Owner 能够理解受众、目的、角色、触发方式、
回复行为和主动发送的边界。Connection、凭证、Gateway、Provider ID 和投递诊断属于
高级集成层。

专门的 Interface Agent 是可选的组织模式，不是必需的 Agent 类型。只有当外部关系
上下文与判断值得一份自己的长期责任时才使用它；否则，一个 Domain Agent 可以直接
持有受治理的外部角色。

## 观察并调整团队

使用 Overview 与 Team 视图来形成问题，而不是形成绩效排名。

- **执行状态**报告什么正在运行、等待、停止或不可用。
- **Inbox 与排队等待**可能反映压力、路由错误、资源占用、Connector 延迟或缺少工具。
- **Capacity** 展示执行与等待的证据；日历上的非执行时间是一个代理指标，并不证明
  某个 Agent 当时可用或不必要。
- **Token 用量**报告消耗，而不是业务价值。
- **Activity** 展示通信证据，而不是权威或组织重要性。

要把信号放在一起解读。高等待伴随低执行，可能说明存在 Goal、重启、权限或
Connector 问题，而不是产能不足。低执行也不足以成为合并一项稀疏但高风险责任的理由。

当一个信号反复出现时：

**当前建议：** 使用下列顺序，以保全证据与决定权归属。

1. 确定受影响的工作和稳定的证据 ID。
2. 询问负责的 Agent，它如何理解这项工作及其边界。
3. 只就相邻负责人自身的经历向其核对。
4. 把任务、工具、调度和 Connector 故障与组织设计问题区分开。
5. 只有在 Owner 确认这是一次长期责任变化之后，才修改 Profile 或关系。
6. 观察之后的真实工作，判断这次调整是否有效。

## 产品边界

CodexLoom 面向高级个人 Owner，包括 One Person Company 经营者，帮助他们使用一支
长期在岗的 Codex Agent Team。它帮助组织这支团队，但不经营 Owner 的整个公司。

CodexLoom 不打算成为：

- CRM、ERP、财务、合同系统或公司 Operating System；
- 通用项目管理或工作流搭建平台；
- 企业多租户管理控制台；
- 自动组织设计器或 Agent 绩效排名系统；
- Codex Runtime、模型智能或 Thread 历史的替代品。

Owner 的目标仍然属于 Owner。Loom 应当减少不必要的配置、查找、转发、跟踪和状态
重建，而由 Agent 完成领域工作。

## 当前限制

- CodexLoom 是本地优先、自托管、处于活跃开发中的项目。
- 部分行为依赖 Codex experimental 接口，可能随 Codex 版本变化。
- 上文描述的 Topic 与 Trigger 在其实现被集成进 `main` 之前需要开发构建。
- 运行指标是诊断证据，且存在已知的数据质量限制；它们无法自动决定组织调整。
- Profile 与已声明的关系表达当前责任，但不会强制每一条通信路由。
- 各外部 Provider 的能力与消息形态不同；Loom 治理身份、权限、凭证、投递和审计，
  同时不抹去 Provider 原生概念。
